# StudyRoomOps - Implementation Assumptions

This document records assumptions made to resolve ambiguities in the source specification.

## Architecture

1. **Project type** is fullstack web (Express + React + MongoDB + OpenCV).
2. **TypeScript** is used for API and frontend for type safety and maintainability.
3. **Python 3.11 + OpenCV** is used for the local vision worker (separate service).
4. **MongoDB replica set** (3 nodes) is required for multi-document transactions used in conflict prevention.

## Business Rules

5. **Default business hours** seed is 07:00-23:00 daily in site timezone (America/Los_Angeles), admin-configurable.
6. **Default reservation duration** policy is 15-240 minutes.
7. **Default no-show grace** is 15 minutes after start; check-in reminder at 10 minutes past start.
8. **Check-in window opens** 15 minutes before reservation start time.
9. **Share links** are authenticated-only internal links, not anonymous public URLs. They expire 24 hours after reservation end.
10. **Creator role** is allowed to manage leads because the specification defines no separate lead-operations role.
11. **Administrator** inherits all Creator and Moderator permissions.

## Membership & Financial

12. **Membership tiers** are admin-configurable and not hardcoded beyond a seeded default "standard" tier.
13. **Reputation tiers** are fixed to New, Trusted, Expert and never grant authorization permissions.
14. **Points redemption** default is 100 points = $1.00 wallet credit; policy-configurable.
15. **Daily financial risk limit** default is $200.00 combined across top-up/spend/refund per user per local calendar day.
16. **Blacklist auto-thresholds** default to 3 no-shows in 30 days or 2 confirmed charge disputes in 180 days.
17. **No external payment processing** included; wallet changes are staff-recorded on-site transactions.

## Security

18. **Argon2id** with memory cost 65536, time cost 3, parallelism 4 for password hashing.
19. **JWT** stored in HttpOnly Secure SameSite=Lax cookie with Mongo-backed session validation.
20. **CSRF** protection via signed token in header required for all mutating requests.
21. **Account lockout** at 5 failed attempts in 15-minute window, 30-minute lockout duration.
22. **Field-level encryption** via AES-256-GCM for sensitive profile data and face embeddings.
23. **File encryption** via AES-256-GCM for stored uploads.
24. **Audit logs** are hash-chained (SHA-256) for tamper evidence. Append-only immutability is enforced at the application layer via `getAppendOnlyCollection()`, which blocks update/delete operations on `audit_logs`. For defense-in-depth, the MongoDB user should also be granted only `insert`+`find` (no `update`/`remove`) on this collection.

## Vision & Access

25. **No physical door-control integration** included; access oversight means monitoring and event generation.
26. **No websocket requirement**; polling is sufficient for reminders and oversight feeds.
27. **Raw video and enrollment images** are not retained after processing; only encrypted embeddings and short-lived event records remain.
28. **Face events TTL** is 30 days, enforced by MongoDB TTL index.

## Content Safety

29. **Sensitive word filter** uses a configurable word list (seeded with defaults), checked server-side on create/update.
30. **Anti-spam** limits: 5 posts per rolling hour, 20 per rolling day, tracked across review creation, question creation, and answers.
31. **On-prem image moderation** is an extension point interface only; no actual implementation required.

## Infrastructure

32. **Docker Compose** for full-stack deployment; no manual package installation required.
33. **LAN HTTPS** via self-signed certificates generated by included script.
34. **No runtime internet dependency** after Docker images are built.
35. **Background jobs** use MongoDB-backed job queue with lease-based processing and retry policy.

## Data Retention

36. Default retention follows spec: face events 30d, sessions 30d, notifications 90d, UGC 2y, reservations/leads 3y, financial/audit 7y.
37. User deletion is soft-delete with anonymization of non-financial data within 30 days.

## Phase Delivery

38. Implementation follows the 6-phase plan defined in the specification.
39. Each phase builds on prior phases without breaking production integrity.
40. Phase placeholder pages are rendered in the UI for upcoming features with clear messaging.
